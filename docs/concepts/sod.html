SoD Segregation of Duties

Função

Identificar combinações de acesso que geram risco

SoD não analisa acessos isolados. Analisa combinações de capacidades. Uma role pode ser completamente justificada isoladamente e, combinada com outra, criar um vetor onde uma identidade executa ações que deveriam exigir aprovação de outra pessoa.

Exemplos de conflito

  • Global Admin + Privileged Role Admin — controle total com auto-escalonamento
  • User Admin + Authentication Admin — cria conta e controla sua autenticação
  • App Admin + Privileged Role Admin — cria service principals com roles críticas

Princípio fundamental

SoD opera em dois momentos:

Design-time: define o catálogo de combinações incompatíveis, independente de usuários.

Run-time: detecta quem possui quais combinações no ambiente real.

Risco mitigado

  • Fraude interna sem separação de função
  • Auto-escalonamento de privilégio
  • Ciclo fechado sem aprovação externa
  • Comprometimento de conta com escopo amplo

Controle ISO

5.3 — Segregation of Duties
Separação de funções conflitantes para reduzir risco de modificação não autorizada, erro ou fraude.

5.3

Papel no modelo: SoD é a camada final de análise de risco antes da decisão. O resultado da análise SoD alimenta diretamente o ciclo de recertificação — cada conflito detectado exige uma decisão formal: remover, manter com exceção documentada, ou reestruturar o acesso.
