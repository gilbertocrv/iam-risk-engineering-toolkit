Data Model

O modelo de dados que conecta as ferramentas do toolkit. Define como dados brutos de acesso são transformados em objetos de risco estruturados — identidade, role, assignment, contexto de privilégio, conflito SoD e risco derivado.

Princípio

Risco em IAM não é um atributo isolado. Ele é derivado da combinação de identidade, acesso (roles e grupos), privilégio (criticidade por tier) e contexto (atividade, MFA, localização). Uma identidade sem role privilegiada é de baixo risco. A mesma identidade com duas roles em conflito SoD, sem MFA e inativa há 120 dias, é risco crítico. O modelo transforma essa lógica em estruturas processáveis.

Entidades principais
Identity 2.1
Entidade autenticável. Ponto de entrada de toda análise — usuário interno, convidado ou conta de serviço.
"id": "uuid", "upn": "user@contoso.com", "type": "internal | guest | service", "status": "active | disabled", "lastSignIn": "datetime"
Role 2.2
Capacidade atribuída. Classifica o acesso por tipo e criticidade — base do modelo RBAC.
"id": "role-id", "name": "User Administrator", "type": "directory-role | group", "classification": "admin | privileged | standard", "iso": ["5.15", "8.2"]
Assignment 2.3
Relaciona identidade com roles. Registra fonte e momento da atribuição — base de correlação entre datasets.
"identityId": "uuid", "roleId": "role-id", "source": "direct | group", "assignedAt": "datetime"
Privilege Context 2.4 · PAM
Define criticidade do acesso por tier. Determina quais controles adicionais são esperados para cada role.
"roleId": "role-id", "tier": "T0 | T1 | T2 | T3", "requiresMFA": true, "isPrivileged": true
SoD Conflict Rule 2.5 · SoD
Define incompatibilidade entre duas roles. Independente de identidade — é o catálogo aplicado em tempo de análise.
"conflictId": "SOD-001", "roleA": "User Administrator", "roleB": "Authentication Administrator", "severity": "critical", "risk": "Cria conta e controla autenticação", "iso": ["5.3", "8.2"]
Risk Object 2.6 · CENTRAL
Unidade central do modelo. Representa uma violação potencial ou real — derivada da combinação de identidade, roles e contexto.
"identityId": "uuid", "roles": ["Role A", "Role B"], "conflictId": "SOD-001", "severity": "critical", "context": { "tier": "T1", "mfa": false, "lastSignIn": "2026-01-10" }
Pipeline de dados

Transformação de dados brutos em objetos de risco estruturados — cada etapa produz a entrada da próxima.

Entrada
roles.csv + groups.csv + signins.csv
Exports do Microsoft Entra ID — dados brutos sem estrutura de risco.
Identity · Role · Assignment (raw)
Etapa 1
RBAC Mapper
Normaliza roles e grupos em objetos Role. Correlaciona por UPN gerando objetos Assignment. Detecta roles sem membros e identidades com múltiplas atribuições.
→ Role objects · Assignment objects
Etapa 2
PAM Classification
Aplica o modelo de tiers sobre os objetos Role. Gera Privilege Context para cada role privilegiada. Enriquece com dados de MFA e inatividade do dataset signins.
→ Privilege Context objects
Etapa 3
SoD Detection
Cruza os Assignments de cada identidade com o catálogo de SoD Conflict Rules. Para cada par de roles que corresponde a um conflito, instancia um Risk Object com contexto de privilégio.
→ Risk Objects · Conflict instances
Saída
Risk Objects estruturados
Conjunto de objetos de risco prontos para decisão — cada um com identidade, roles envolvidas, conflito, severidade e contexto completo.
→ findings.csv · manifest.json
Outputs por camada

Inventário — RBAC

  • Roles classificadas por tipo
  • Grupos por criticidade
  • Identidades e atribuições
  • Roles e grupos órfãos

Privilégio — PAM

  • Contas por tier (T0–T2)
  • Ausência de MFA
  • Inatividade acima do limiar
  • Contas de serviço sem controle

Conflitos — SoD

  • Lista de violações por identidade
  • Severidade por conflito
  • Risco derivado da combinação
  • Mapeamento ISO por regra

Sustentabilidade

  • Roles e grupos órfãos
  • Identidades sobrecarregadas
  • Acumulação acima do limiar
  • Grupos com volume atípico
Objetivo do modelo

O modelo transforma dados brutos de acesso em informação estruturada de risco. Cada Risk Object é auditável, rastreável e diretamente acionável no ciclo de recertificação — sem subjetividade, sem análise ad hoc.
